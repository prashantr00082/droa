# Deep Repository Ontology Agent (DROA)
**An Enterprise Knowledge Layer for Agentic AI**

## 1. Executive Summary
The **Deep Repository Ontology Agent (DROA)** is an autonomous, multi-agent system designed to ingest any software repository and produce a structured, three-layer **Repository Knowledge Base (RKB)**. 

Unlike standard retrieval-augmented generation (RAG) pipelines that simply chunk and embed raw source code, DROA builds a true **Enterprise Knowledge Layer (EKL)**. It extracts the implicit semantic meaning of a codebase—its concepts, flows, ownership, and architectural patterns—and packages it into a highly structured Knowledge Graph. This ensures that downstream coding agents (or human developers) can query the RKB to understand *why* the code exists and *how* it fits together, drastically reducing hallucinations and "semantic drift" in enterprise AI deployments.

---

## 2. What Makes DROA Unique?

### A. Context Economy & Layered Abstraction
Most AI coding assistants fail at scale because they try to shove the entire repository into a single context window, or retrieve random snippets of code. DROA solves this by enforcing a strict hierarchy of abstraction:
1. **Ontology (The Map)**: High-level business concepts, directory structures, and request flows. (No code implementation is read here).
2. **Architecture (The Blueprints)**: Logical subsystems, data models, and cross-module dependencies.
3. **Implementation (The Details)**: Deep, structural documentation of individual files.

Agents querying this system read the Ontology first, zeroing in only on the Implementation details they actually need.

### B. The "Cite-or-Drop" Rule (Hallucination Prevention)
To ensure absolute trust in the generated knowledge base, DROA enforces a strict **Validator mechanism**. Every single architectural claim or conceptual definition generated by an LLM *must* cite the specific file, function, or AST node it was derived from. If an LLM hallucinates a concept without proof, the Validator rejects it and re-queues the task.

### C. Reciprocity Checking
When documenting complex codebases, dependency claims often drift. If Module A claims to depend on Module B, DROA’s deterministic `Reciprocity Checker` verifies that Module B acknowledges being used by Module A. Any mismatches are flagged in an automated `reciprocity_report.md`.

### D. Native GraphRAG Integration
Instead of exporting flat text for a Vector Database, DROA exports its structured JSON sidecars into a **Neo4j Cypher Script**. This transforms the codebase into a queryable Knowledge Graph, mapping Modules to Symbols, and Symbols to their Dependencies.

## 3. Core Components & Workflow

DROA is orchestrated using a **LangGraph State Machine** backed by a robust SQLite persistence ledger (`.rkb/state.db`). This guarantees that the process is atomic, resumable, and supports incremental updates (only processing files whose `sha256` hashes have changed).

The pipeline consists of the following phases:

### Phase 1: Scanner & Orchestration (Conductor)
The **Conductor** manages the workflow. It scans the repository, hashes the files, and populates the SQLite task queue. It uses a `BEGIN IMMEDIATE` transaction lock to ensure thread-safe, concurrent task claiming.

### Phase 2: Ontology Extraction (Cartographer)
The **Cartographer** agent acts as the surveyor. It never reads function implementations; it only looks at AST symbols, directory trees, and `CODEOWNERS` files. It produces:
- `concepts.md`: The core domain vocabulary (e.g., "What is a Task?").
- `flows.md`: Inferred request and data flows.
- `organization.md`: Codebase ownership mapping.

### Phase 3: Deep Documentation (Module Documenter)
For every Python file, the **Module Documenter** relies on a Tree-sitter powered AST parser to extract raw structural truths (classes, functions, imports). The LLM is then asked to explain *only* the responsibilities of those predefined nodes, outputting highly structured JSON sidecars alongside Markdown.

### Phase 4: Quality Gates (Validator)
The **Validator** scores the output of the Module Documenter based on API coverage, formatting, and the Cite-or-Drop rule. If a document scores below 75% confidence, it is automatically re-queued for another pass.

### Phase 5: Synthesis (Synthesizer)
Running at the very end, the **Synthesizer** ingests the Cartographer's high-level ontology and the detailed JSON sidecars to generate macroscopic architecture documents:
- `components.md`: Groups modules into logical subsystems.
- `data-models.md`: Aggregates all Data Entities (Pydantic, SQLAlchemy) into a single dictionary.
- `lessons_learned.md`: Rolls up architectural decisions from an append-only Agent Memory ledger.

### Phase 6: GraphRAG Export
The final step translates all validated JSON structural data into `load_graph.cypher`, making the entire repository ready for ingestion into Neo4j.

## 4. The Value Proposition

By deploying DROA in your enterprise, you achieve:
1. **Faster AI Onboarding**: New coding agents don't have to guess what your proprietary data models mean. They query the Knowledge Graph.
2. **Human-Readable Documentation**: Generates beautiful, perfectly accurate Markdown documentation that human engineers actually want to read.
3. **No Stale Knowledge**: Because the system is incremental and backed by a SQLite ledger, you can run DROA as a CI/CD job. It only updates the documentation for files that changed in the latest commit, keeping the Knowledge Base perfectly in sync with the source code.
